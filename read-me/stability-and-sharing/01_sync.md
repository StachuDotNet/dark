# Push/Pull Sync Mechanism

## Model
Works like the existing local SCM — rebases and merges — but between a local client and the central server. Client/server, not P2P.

The server's DB is the canonical "remote". Each client has a full local copy. Sync is explicit (`dark push` / `dark pull`), never automatic.

## What syncs

**Key insight from code review:** Only ops and commits need to travel over the wire. When ops are inserted via `insertAndApplyOps`, the playback system (`PackageOpPlayback.applyOps`) automatically creates/updates content items, locations, and dependencies. This is a MASSIVE simplification — no need to sync content tables, location tables, or dependency tables separately.

- **Committed ops + commits**: yes — this is ALL that needs to transfer
- **Content items** (package_types/values/functions): **auto-generated** by op playback on receiving side
- **Locations**: **auto-generated** by op playback
- **Dependencies**: **auto-generated** by op playback
- **Branches**: yes (branch metadata for new branches)
- **WIP ops**: **no** — WIP is local-only, never pushed (`commit_id IS NULL`)
- **user_data**: **no**
- **Traces**: **no**

## ID Model (from code review)

Understanding this is critical for sync correctness:

- **Op IDs are content-addressed**: SHA256 hash of binary-serialized op → first 16 bytes → GUID (`Inserts.fs:computeOpHash`). Same op content → same ID on any machine. `INSERT OR IGNORE` skips duplicates.
- **Package item IDs are location-addressed**: IDs come from name lookup (`PackageIDs.Type.idForName owner modules name`), NOT from content hashing. Darklang-owned items get hardcoded UUIDs from a static map; custom packages get `Guid.NewGuid()`.
- **Why this matters**: Content-addressed ops mean sync won't create duplicates. Location-addressed items mean renaming doesn't break references (ID stays stable for a given name). The two-pass parsing with `stabilizeOpsAgainstPM` ensures IDs are deterministic for the same `.dark` files.

## Wire Format

**Ops as JSON over HTTP** — simpler than originally planned:

The BuiltinPM already converts ops between F# `PackageOp` types and Darklang values (via `PT2DT.PackageOp.toDT`/`fromDT`). Darklang values serialize to JSON. So the simplest wire format is **JSON**.

**What travels per push/pull:**
1. Commit records (id, message, branch_id, created_at)
2. Ops as Darklang values (JSON-serialized `List<PackageOp>`)
3. That's it — content items, locations, and dependencies are auto-generated by op playback

**Data sizes (from current DB):**
- Avg op: 327 bytes (binary), probably ~500-800 bytes as JSON
- Typical branch push: ~100 ops = ~50-80KB JSON
- Full DB (for clone): 12MB as SQLite file

For clone, binary (download the `.db` file directly) is better. For incremental push/pull, JSON is simple and fast enough.

**If JSON is too slow later:** The ops also exist as binary blobs (`op_blob` column in `package_ops`). Could switch to binary transport, but JSON is the right starting point — easier to debug, and the data volumes are tiny.

## Push

```
dark push
```

Pushes the current branch's committed ops to the server.

Preconditions (mirror the local merge preconditions):
- Branch must be up-to-date with server's version of the same branch (pull first if behind)
- No WIP on the branch (commit or discard first)

Flow:
1. Client asks server: `GET /branches/:id/commits?since=:last_known_commit`
2. If server has commits client doesn't → reject, must pull first
3. Client computes diff: commits/ops it has that server doesn't
4. Client sends: `POST /branches/:id/ops` with new commits + ops (as JSON)
5. Server validates and applies them (reuses existing `insertAndApplyOps` — content tables, locations, and deps are auto-generated by op playback)
6. Server confirms success

If the server has commits the client doesn't (someone else pushed), push is rejected — client must `dark pull` and rebase first. Same discipline as `git push` requiring an up-to-date branch.

### Pushing a new branch
If the branch doesn't exist on the server yet, push creates it. The branch's parent must already exist on the server.

## Pull

```
dark pull
```

Pulls the latest committed state from the server for the current branch.

Flow:
1. Client asks server: `GET /branches/:id/ops?since=:last_known_commit`
2. Server sends new commits + their ops (as JSON)
3. Client applies them locally using `insertAndApplyOps` (content tables, locations, and deps are auto-generated by op playback)

If the client has local commits that the server doesn't (diverged), pull flags it — client needs to push first, or rebase.

### Pulling a branch that doesn't exist locally
`dark pull --branch feature-x` creates the branch locally and downloads its full history.

## Clone

```
dark clone <server-url>
```

Bootstrap a fresh local install from the server. Downloads the server's `data.db` (or a snapshot of it) wholesale via `GET /snapshot`. No incremental sync — just grab the whole thing.

This is the escape hatch for "I need a clean start" and also the initial setup path for a new machine.

**Implementation:** `Stdlib.HttpClient.get(serverUrl ++ "/snapshot", [])` → write response body to `rundir/data.db`.

## SCM Invariants for Sync

From code review of `Merge.fs`, `Rebase.fs`, `Branches.fs`:

### Merge preconditions (must hold across machines)
- Branch must be rebased (`base_commit_id` == parent's latest commit)
- Branch must have no WIP
- Branch must have no active children
- Branch must have at least 1 commit
- Cannot merge main

**Sync implication:** Merge should only happen on the server (or locally then pushed immediately). If a merge happens locally without pushing first, the parent branch state on the server may be different, causing location deprecation mismatches.

### Rebase behavior
- Updates `base_commit_id` to parent's latest commit — **does NOT rewrite ops**
- Validates no location conflicts (same path modified on both branch and parent since `base_commit_id`)
- Conflicts reported by location path; user must fix manually

**Sync implication:** Rebase is safe across machines because it doesn't modify op content. The `base_commit_id` pointer update is the only state change. But both machines must agree on what "parent's latest" is — which push/pull ensures.

### Recommended workflow for sync + merge
1. `dark pull` (on your branch — get any ops your coworker pushed)
2. `dark pull --branch main` (get latest main)
3. `dark rebase` (local branch onto updated main)
4. `dark push` (push rebased branch to server)
5. `dark merge` (merge into parent — this should happen on the server, or locally then immediately pushed)

### Key constraint: merge should be atomic with push
If you merge locally, the merge ops (moving ops to parent branch, deprecating locations) must be pushed to the server atomically. Otherwise the server's state diverges. Options:
- **Merge on server**: `dark merge --remote` sends a merge request to the server, which performs the merge
- **Merge locally then push all**: merge locally, then push both the child branch changes AND the parent branch changes in one operation

## Branch Ownership

No branch locking. Convention: each person works on their own branches, merges through `main` on the server. If two people push to the same branch, last-to-push must pull+rebase first (same as git).

## Server HTTP API

```
GET  /branches                          → list branches (JSON)
POST /branches                          → create branch (JSON: name, parent_id)
GET  /branches/:id/commits?since=:id    → commits since X (binary)
POST /branches/:id/ops                  → push ops (binary)
GET  /branches/:id/ops?since=:commit    → pull ops since commit (binary)
POST /branches/:id/merge                → merge branch on server (JSON)
GET  /snapshot                           → download full data.db (binary)
GET  /snapshots                          → list available backup snapshots (JSON)
GET  /snapshots/:name                    → download a specific snapshot (binary)
```

Op payloads travel as JSON (via PT2DT → Dval → JSON roundtrip). Branch/commit metadata is also JSON. Clone (`GET /snapshot`) is the only binary endpoint (raw `.db` file download).

## Git Branch ↔ Darklang Branch Correspondence

When working on a git branch of Dark itself (e.g., `git checkout add-new-expr-variant`), you might want your local Darklang environment to use a specific Darklang branch (so your package code changes are isolated).

Options:
- **Env var**: `DARK_BRANCH=pr/1234 dark fn ...` — overrides the default branch for that CLI session
- **Config file**: `.darklang-branch` in the repo root (gitignored) — local dev sets this per git branch
- **Convention**: Darklang branch names match git branch names (or `pr/<number>`)
- **CLI flag**: `dark --branch pr/1234 fn ...` (this already exists per the docs)

The `--branch` flag already exists. The missing piece is a way to set a default branch for a working directory so you don't have to pass it every time.

Likely: an env var (`DARK_BRANCH`) or a local config file that the CLI reads on startup to set `currentBranchId`.
