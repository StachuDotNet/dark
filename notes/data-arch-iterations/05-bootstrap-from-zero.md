# Iter 05 — bootstrap from absolute zero

The networking doc has the login flow ("opens browser, polls,
gets token, writes config"). The rewrite doc has `dark init` and
`dark sync clone tail-major`. Neither answers: **after I download
the binary, how many seconds until I can write `let x = 1`?**

This iteration is the timeline.

## The user's first hour

```
00:00   I download `dark` (~80 MB AOT'd binary, single file).
00:05   `chmod +x dark && ./dark`
00:06   ←  ?  ←
```

What happens at 00:06 sets the entire mental model. Three options:

A. **Aggressive:** auto-init with embedded snapshot, drop into
   the REPL with stdlib loaded. User sees `λ ` in 2 seconds.
   Login is later.

B. **Conservative:** ask first ("first run? want to set up?"),
   require login before any state. User sees a wizard.

C. **Hybrid:** auto-init silently (snapshot extracts; daemon
   builds main projection in the background). Surface a banner
   "you're set up locally; `dark login` to enable sync." First
   command works immediately against the embedded stdlib.

I'd take C. Reasons:

- Online-only setup is an anti-feature (planes, weak wifi, dev
  containers without external network).
- A REPL prompt with no functions is a worse experience than a
  REPL with stdlib but no account.
- The login-later flow lets the user *see* the product before
  deciding to give it an email.

So:

```
00:06   $ dark
00:06.1 First run detected. Initializing ~/.darklang/…
00:06.2 [extracting embedded snapshot: 47 MB → ops.db (52 MB)]
00:06.6 [building main projection: ~50K stdlib ops]
00:09.4 ✓ Ready (3.4s).

        Tip: run `dark login` to sync to other devices and back up
             your work to dark.run.

        $ dark help
        ...
        $
```

3-4 seconds to first prompt. Most of that is the projection
build. Snapshot extract is ~400ms (gzip-decompress to disk).

## What's in the embedded snapshot?

Two extreme positions:

1. **Everything Dark ships with.** Every package fn ever written
   in the official tree, all 30K-50K op rows in `ops.db`,
   plus blobs.
2. **Just `Darklang.Stdlib`.** A few thousand ops; users pull
   `Darklang.LanguageTools` etc. on demand from the hub.

(1) gives a fully-functional CLI with no network. (2) makes the
binary tiny and the upgrade path simple — but every fresh
install needs the hub.

I'd ship the snapshot at the **"useful offline"** point: stdlib +
languageTools + the agent module + the CLI's own package code.
Maybe 100K ops. Big enough that the LSP works offline; small
enough to fit in ~30-50MB compressed.

The snapshot is regenerated by CI from the `main` branch's `ops`
table. Each release of `dark` ships with a fresh snapshot. The
embedded version's hash is in `ops.db` itself
(`device_identity.bootstrap_hash`); the user's first sync pulls
diffs since that hash.

## The snapshot binary format

Same `ops.db` schema as runtime. Compressed at build time
(gzip or zstd; SQLite blobs already compress well, but the file
is 60% repeated UTF-8 in `stream` and `key` columns — zstd
crushes that). Embedded as a resource in the binary.

On first run:

```fsharp
let bootstrapFromEmbedded () : Task<unit> = task {
  use stream = Assembly.GetManifestResourceStream("ops-snapshot-v0.db.zst")
  use decompressed = new ZStdDecompressionStream(stream)
  let path = Config.opsDbPath
  use file = File.Create(path)
  do! decompressed.CopyToAsync(file)
  // Mark snapshot version so we know what to delta against on sync.
  use conn = Sqlite.connect Sqlite.OpsRW
  Sql.query "INSERT INTO bootstrap (version, extracted_at) VALUES (?, ?)"
    conn [version; nowMs] |> Sql.executeStatementSync
}
```

~400ms for a 50MB compressed → 200MB uncompressed snapshot on
modern hardware. Fast enough that we don't need a progress bar
unless we get above ~1s.

## The first projection build

After snapshot extraction, the daemon needs `pkg.db` for `main`
before any user command works.

The slow case today: 50K ops × ~50µs/op for projection apply =
2.5s. Acceptable for a first-run wait. Not acceptable for every
launch.

Cached: subsequent launches see `~/.darklang/projections/main/
pkg.db` already exists; daemon catches up incrementally
(usually zero new ops since last shutdown). Boot is sub-100ms.

## The login flow, revised

Per the networking doc, but with the bootstrap context:

```
$ dark login
Opening browser to https://dark.run/cli-auth?code=4f8b
[user clicks Authorize on dark.run]
✓ Logged in as stachu@dark.run on instance "stachu-laptop"

  • daemon: connecting to wss://hub.dark.run/instance … ✓ (220ms)
  • sync:   pulling user-private ops … ✓ (1.2s, 387 ops)
  • build:  catching up projections (main, feat-cli) … ✓ (340ms)

You're synced. Try `dark devices` to see your other instances.
```

End-to-end timing for an existing-account user on a new device:

| Phase | Duration |
|---|---|
| `dark login` browser flow | ~20s (human) |
| Hub WS connect after token | ~200ms |
| Pull user-private ops since snapshot hash | 0.5–10s (depends on history) |
| Catch up projections | 0.1–1s (per branch) |
| First sync round-trip | ~300ms |

Total under-a-minute end-to-end. The 20s human portion dominates;
the rest sums to seconds.

For a **new** account, `dark login` opens a signup form first;
add ~30s for the user to type email + password / OAuth. Still
under 90s end-to-end.

## What the user can do *during* sync clone

Critical UX question. Sync pulls might be 10s on a slow
connection. The user has the embedded stdlib available
*immediately* — they shouldn't have to wait for a full clone to
write `let x = 1`.

Solution: **the snapshot is a fully-functional `ops.db`.** The
daemon serves commands against it from millisecond zero. Sync
clone happens in the background; daemon merges new ops into
`ops.db` and bumps the projection. The user sees it as:

```
$ dark login
[opens browser, …, success]
✓ Logged in. Syncing in background.

$ dark fn Darklang.Stdlib.List.map     # works immediately, against snapshot
[editor opens]

[10 seconds later, in another terminal]
$ dark sync status
Syncing: 387 of 1,204 ops pulled (32%). ETA 15s.
```

When sync finishes, the user gets a notification in the next
command prompt:

```
$ dark fn Some.Other.Fn
✓ Sync complete. 1,204 ops pulled, 12 conflicts detected.
  Run `dark conflicts` to view.
```

## Failure modes

### Network down during login

```
$ dark login
Opening browser to https://dark.run/cli-auth?code=4f8b
✗ Could not reach hub.dark.run (timeout after 10s).

You can keep using Dark offline against the embedded snapshot.
Run `dark login` again when you have network.
```

The snapshot is still functional. User loses sync for now.
`dark daemon status` shows "offline; will retry every 60s on
next command."

### Hub up but instance token rejected

```
$ dark login
Opening browser to https://dark.run/cli-auth?code=4f8b
✗ Authorization failed: code expired.
  Run `dark login` to retry.
```

Codes expire after 5 minutes (server-side). User retries.

### Embedded snapshot corruption

Rare but should be detected. On extract, hash the file and
compare against the manifest. If wrong:

```
$ dark
First run detected. Initializing ~/.darklang/…
✗ Embedded snapshot integrity check failed.
  This binary may be corrupted. Re-download or run `dark --skip-bootstrap` to skip
  (you'll need to `dark sync clone <peer>` to get any packages).
```

Recoverable with a re-download. `--skip-bootstrap` is the manual
override.

### Network died mid-clone

Resumable. Sync watermarks track the last-pulled op hash; restarting
`dark sync` resumes from that point. The clone is a normal sync
pull, just larger.

```
$ dark login
…
[wifi dies after 800 of 1,204 ops]

$ dark daemon status
Sync: incomplete (804/1,204). Last pulled: ab47…
Will resume on next sync attempt or manually with `dark sync`.

$ dark sync
Resuming from ab47… → done (400 ops, 6s).
```

The projection is in an OK state for the 804 ops; pulling 400
more catches it up.

### Daemon crashes during projection build

Projection build writes a `building` flag in `projection_state`
before starting. On startup, daemon checks: if flag is set, the
projection is incomplete; delete and rebuild from `ops.db`. No
data loss because `ops.db` is the source of truth.

```
$ dark
Starting daemon…
Detected incomplete projection for branch `main`. Rebuilding…
✓ (3.4s)
```

### `~/.darklang/` corrupted (disk problem)

Last resort. `dark recover` re-extracts the snapshot, attempts to
catch up from hub. If `ops.db` itself is corrupted, the user loses
local-only state (uncommitted WIP) but everything synced is
recoverable from the hub.

## Re-bootstrap

User intentionally wants to start over (e.g., debugging, or
freed up disk space):

```
$ dark reset --confirm
Stopping daemon…
Deleting ~/.darklang/projections/ (180 MB)
Deleting ~/.darklang/apps/ (60 MB)
Re-extracting embedded snapshot…
✓ Reset complete. Run `dark login` to re-sync your work.
```

`ops.db` itself is preserved by default (it's the source of
truth). `--full` deletes `ops.db` too.

## Multi-instance bootstrap on the same machine

Power user wants two configs (e.g., personal + work):

```
$ DARK_ROOT=~/dark-work dark login
$ DARK_ROOT=~/dark-personal dark login

# Aliases in shell:
alias darkw='DARK_ROOT=~/dark-work dark'
alias darkp='DARK_ROOT=~/dark-personal dark'
```

Each `DARK_ROOT` gets its own daemon (different sockets,
different ops.db). Bootstrap runs once per `DARK_ROOT`. No
contention.

## The control flow, code-shaped

```fsharp
module LocalExec.Bootstrap

let main args = task {
  match Config.darkRoot |> Path.exists with
  | false ->
    // First run, totally fresh. Wizard or auto-init.
    do! firstRunInit ()
  | true ->
    // Existing config, just start daemon.
    ()

  // Connect or spawn daemon. From now on everything is
  // RPC over the unix socket.
  let! daemonClient = Daemon.connectOrSpawn ()
  let! result = daemonClient.RunCommand args
  return result.exitCode
}

and firstRunInit () = task {
  Directory.CreateDirectory(Config.darkRoot)
  printfn "First run detected. Initializing %s…" Config.darkRoot

  // Extract embedded snapshot.
  do! Snapshot.extractEmbedded Config.opsDbPath

  // Spawn daemon, which will build main's projection on first request.
  // We don't wait for it — the next command (`dark help` etc.) will
  // trigger projection build lazily.
  do! Daemon.start ()

  printfn "✓ Ready. Run `dark login` to sync your work."
}
```

Total path: ~400ms for snapshot extract + ~100ms for daemon
spawn + projection build deferred to first real command. The
"Ready" message appears in well under 1s.

## Daemon-as-installer thinking

A nice property: the daemon can update *itself* once it's
running. The hub knows what versions exist:

```
$ dark daemon status
Daemon version: 0.4.2
Latest: 0.4.5

$ dark daemon update
Downloading darkd 0.4.5 (37 MB)…
Verifying signature… ✓
Restarting daemon… ✓
$ dark daemon status
Daemon version: 0.4.5
```

The embedded snapshot in 0.4.5 might be newer than what's in
0.4.2's ops.db. Update flow:

1. New binary downloads.
2. New snapshot extracted to `~/.darklang/cache/snapshot-v0.4.5.db`.
3. Sync the diff between old snapshot's last hash and new
   snapshot's last hash into `ops.db`.
4. Daemon restarts using new binary.

This means upgrades don't require network — the new binary
contains everything needed. Sync just keeps the user's *private*
state current.

## Bootstrap as Dark code (per iter 02)

Per the F#-vs-Dark line, `firstRunInit` and `daemonClient.
RunCommand` should mostly be Dark. F# does:

- File extraction (`Snapshot.extractEmbedded` — ~30 lines).
- Process spawning (`Daemon.start` — ~20 lines).
- Unix socket client (`Daemon.connectOrSpawn` — ~50 lines).

Everything else is `Darklang.Bootstrap.*`:

```dark
module Darklang.Bootstrap

let firstRunInit () : Stdlib.Result.Result<Unit, String> =
  // Extract snapshot via Builtin.snapshotExtract
  match Builtin.snapshotExtract Config.opsDbPath with
  | Error e -> Stdlib.Result.Result.Error e
  | Ok _ ->
    Stdlib.printLine "✓ Ready. Run `dark login` to sync."
    Stdlib.Result.Result.Ok ()

let onLogin (token: String) (instanceId: Uuid) : Unit =
  Stdlib.printLine "Logged in. Syncing in background."
  // Fire-and-forget background task
  Stdlib.Task.spawn (fun () ->
    let result = Sync.cloneFromHub token instanceId
    match result with
    | Ok stats ->
      Stdlib.printLine $"✓ Sync complete. {stats.opsPulled} ops pulled."
    | Error e ->
      Stdlib.printLine $"⚠ Sync incomplete: {e}")
```

CLI strings, error messages, the "first run" experience — all
Dark, all readable by the LSP and editable by the user.

## Open questions

1. **What about ARM64 / different architectures' snapshots?**
   Same snapshot bytes for every arch (ops.db is portable). One
   snapshot, multiple binaries. ✓.

2. **Snapshot age cliff.** A user installs a 6-month-old
   binary. Snapshot's last op is ancient. Sync pull is huge.
   Acceptable; we shouldn't optimize for "people running old
   binaries" — the upgrade path is `dark daemon update`.

3. **Self-hosted hub.** If a team runs their own hub at
   `hub.team.example`, where does that config go? `dark login
   --hub https://hub.team.example`. Token + hub URL stored
   together in `config.toml`. Hub is just a URL.

4. **Bootstrapping without an embedded snapshot.** Some builds
   (slim AOT for resource-constrained environments) might ship
   without the snapshot. Then `dark` requires login on first
   run. CLI flag: `--minimal-binary` reports this; bootstrap
   says "this binary doesn't include the snapshot; please log
   in or `dark sync clone <peer>` to bootstrap."

5. **What happens when you run `dark` before `dark login` on
   first install?** Per the design above, snapshot is enough
   for read-only operation. Writes are buffered locally until
   login. Or: refuse writes until login, with a clear error.
   Probably the latter is simpler — the auth gate from iter 03
   already requires a non-None accountID for ops to be append-
   able.

6. **The `darkd` daemon vs `dark` CLI version skew.** Daemon
   running 0.4.2; user runs `dark` from a 0.4.5 binary. The CLI
   should detect this and refuse-or-warn ("daemon is older;
   restart or `dark daemon update`"). Cheap to implement; small
   protocol-version handshake.

## TL;DR

First run: 3-4s to ready, snapshot embedded in the binary.
Login is later, optional, ~20s human + ~5s machine. All-online
or all-offline both work; mixed states (snapshot + partial
sync) are normal. Daemon owns the lifecycle; CLI is a thin RPC
client. Re-bootstrap is `dark reset`.

The product story: **download → run → see a prompt in 4 seconds
→ log in if you want sync.** That's the whole bootstrap.
