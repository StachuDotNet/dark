# meta-reflections/

How the *process itself* is going — the recurring loop that produces this work.
This feeds back into design: lessons here shape how the next loop is run, what
the reflection template asks for, and where to put external stop signals.

Scope: these files are about *making the work*, not the work's content. Per-
project results live in `results/`; design decisions live in `design/`;
backlog items live in `issues-and-improvements/`. Anything that's really one of
those was routed there instead of here.

## Files

- [what-the-loop-is-good-at.md](what-the-loop-is-good-at.md) — the loop
  behaviours that consistently earned their cost: live probing over vibes,
  closing old questions one at a time, falsifiable predictions, move-type
  variety as a governor, synthesis once content exists.
- [where-the-loop-struggles.md](where-the-loop-struggles.md) — the failure
  modes: diminishing returns it won't stop on its own, narration that outlives
  its use, inconclusive probes, sticky over-claims, low-insight single-lane
  stretches, the fuzzy "out of work" line (there is always more mechanical
  work), and scaffolds built ahead of their data.
- [process-risks.md](process-risks.md) — what can quietly corrupt the output:
  self-grading skew, false completion, doc-vs-ground-truth drift, cross-
  reference rot, throughput mistaken for progress.

## The one rule everything here points at

The loop is good at finding the next thing to do and bad at knowing when to
stop. Every guard above resolves to the same move: keep the success signal
*external* to the loop — a probe that ran, an artifact that passed, a question
that closed — never the loop's own assertion that it made progress.
